## Scope Revision — Lane B (2026-05-14)

The original Lane B (MDX migration) was scoped against a greenfield assumption. The actual codebase already has a working typed-TS content pipeline: 53 entries across `lib/services-data.ts` (18), `lib/industries-data.ts` (11), `lib/blog-data.ts` (11), `lib/guides-data.ts` (10), `lib/solutions-data.ts` (3), each with rich nested interfaces (FAQs, process steps, related items, SEO meta). Existing `[slug]` routes already use `generateStaticParams()` and inject FAQ/Breadcrumb JSON-LD.

Full MDX migration would mean rewriting ~50,000 lines of structured TS to MDX, refactoring every page-content component, and taking real risk against a live deploy — not Phase 1 shaped.

**Revised Lane B: formalize the existing contract with Zod.** Adds `lib/content-schema.ts` (Zod schemas matching the existing interfaces) + `scripts/validate-content.ts` (build-time validator, wired as npm prebuild). Build fails on structural drift; soft warnings (e.g. meta description length) surface SEO issues without blocking.

D2 (MDX in repo) is deferred. May revisit for case studies or other new content types in a future PR. The existing typed-TS pattern stays.

**Initial validator output:** all 53 entries pass structural validation. 15 soft warnings (meta descriptions over 160 chars on industries, blog posts, and one guide) — not blocking, captured as content cleanup TODO.

### MDX content pipeline

```
build time:
  /content/services/*.mdx
        │
        ▼
  lib/mdx.ts ── reads frontmatter (Zod-validated) + body
        │
        ▼
  app/services/[slug]/page.tsx ── generateStaticParams() for all slugs
        │
        ▼
  static HTML in .next/server/app/services/<slug>.html
        │
        ▼
  Vercel CDN
```

Every `[slug]` route uses `generateStaticParams()` so all pages are pre-rendered at build. Zero SSR for marketing content. Lighthouse-friendly by default.

### Frontmatter schema (Zod) — service page example

```typescript
// lib/content-schema.ts
export const ServiceFrontmatter = z.object({
  title: z.string(),
  slug: z.string(),
  description: z.string().min(120).max(160),   // for <meta name="description">
  audience: z.enum(['founders', 'sme', 'both']),  // drives cross-links
  ogImage: z.string().optional(),
  publishedAt: z.string().datetime(),
  updatedAt: z.string().datetime().optional(),
  relatedServices: z.array(z.string()).optional(),
  relatedIndustries: z.array(z.string()).optional(),
});
```

Same pattern for `IndustryFrontmatter`, `CaseStudyFrontmatter`, `BlogFrontmatter`. Build fails loudly if any MDX file violates schema — no silent SEO regressions.

### SEO foundation

| Surface | Implementation |
|---------|---------------|
| `<title>` per route | Next.js `generateMetadata()` per page |
| `<meta name="description">` | from MDX frontmatter `description`, validated 120–160 chars |
| Canonical URL | `metadata.alternates.canonical` per page |
| Open Graph image | Per-route in `/public/og/`, or generated via `@vercel/og` from title + brand |
| `sitemap.xml` | `app/sitemap.ts` — enumerates all static + dynamic routes |
| `robots.txt` | `app/robots.ts` |
| JSON-LD structured data | `Organization` schema on `/`, `Service` on each service page, `Article` on blog posts, `BreadcrumbList` on deep pages |
| Heading hierarchy | Single `<h1>` per page, semantic `<h2>`/`<h3>` for sections |
| Image alt text | enforced via TypeScript `alt: string` (non-optional) on a wrapper `<Img />` component |

## Code Quality

### Component reuse

Every reusable section becomes a typed component with content as props. Pages assemble sections; they don't reimplement layouts.

**Bad (what the current Vercel site likely does):**
```tsx
// app/page.tsx
<section className="...">
  <h2>50+ Projects</h2>
  ...
</section>
```

**Good (what we're enforcing):**
```tsx
// app/page.tsx
<MetricsRow metrics={metrics} />

// components/sections/MetricsRow.tsx
type Metric = { value: string; label: string; };
export function MetricsRow({ metrics }: { metrics: Metric[] }) { ... }

// content/data/metrics.ts
export const metrics: Metric[] = [
  { value: '50+', label: 'Projects shipped' },
  { value: '8-16 wks', label: 'MVP to launch' },
  { value: '98%', label: 'On-time delivery' },
];
```

**DRY rule:** if a value appears on more than one page (metrics, client logos, team, testimonials), it lives in `content/data/*.ts` as a typed array with a single source of truth.

### Type safety for MDX

`lib/mdx.ts` parses each MDX file through its Zod schema before returning. Frontmatter typo? Build fails. Wrong audience enum? Build fails. Missing description? Build fails. We catch SEO regressions at build, not in production.

### Naming conventions

- Page components: `page.tsx` (Next.js convention).
- Section components: PascalCase, descriptive, in `components/sections/`.
- Data files: kebab-case singular for content (`team.ts`), PascalCase for components.
- MDX slugs: kebab-case, match the URL slug exactly.

## Performance

| Concern | Approach |
|---------|----------|
| Initial load | All pages static at build time. Zero runtime DB calls. |
| Image optimization | `next/image` with sized variants. All images served as AVIF/WebP. Hero images preloaded. |
| Font loading | `next/font` with `display: swap`. Self-hosted fonts. No FOIT. |
| Bundle size | RSC by default, `'use client'` only on interactive sections (carousels, audience picker). Target initial JS < 80KB compressed. |
| Lighthouse targets | Performance ≥ 90, Accessibility ≥ 95, Best Practices ≥ 95, SEO = 100 per page |
| Core Web Vitals budget | LCP < 1.8s, INP < 200ms, CLS < 0.05 |
| Calendly embed | Lazy-load only when user reaches CTA section (intersection observer) — Calendly is the biggest perf cost on the current site |

**Caching:** marketing site has no user state. ISR not needed for Phase 1; full static is simpler. If/when blog posts get frequent updates, switch to ISR with `revalidate: 3600`.

| Codepath | Failure | Test? | Handled? | UX outcome |
|----------|---------|-------|----------|------------|
| Calendly embed | Calendly is down | No (out of scope) | Partial | Fallback link to `mailto:` or contact form. **Critical gap if not handled.** |

## Worktree Parallelization

Three independent lanes can run in parallel:

```
Lane A — Homepage + nav + audience doors        [touches: app/, components/sections/, components/nav/]
  └── Sequential: 1) build sections → 2) wire homepage → 3) build /founders + /ai-for-sme → 4) update nav

Lane B — MDX infrastructure + service/industry/blog content pipeline   [touches: lib/, content/, app/services/[slug], app/industries/[slug], app/blog/[slug]]
  └── Sequential: 1) lib/mdx.ts + Zod schemas → 2) [slug] routes → 3) write initial MDX content

Lane C — SEO foundation + /about stats fix      [touches: app/sitemap.ts, app/robots.ts, lib/seo.ts, app/about/]
  └── Sequential: 1) sitemap + robots → 2) metadata helpers → 3) per-page metadata → 4) /about stats fix
```

**Launch order:**
1. Launch A, B, C in parallel worktrees. None share modified directories (A=homepage+nav, B=mdx+content, C=seo+about).
